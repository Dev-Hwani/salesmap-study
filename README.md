# salesmap-test

세일즈 CRM 도메인을 이해하고  
구조적인 설계와 명확한 책임 분리를 목표로 개발한 CRM 과제 프로젝트입니다.

단순 기능 구현이 아닌  
“왜 이런 구조와 기술을 선택했는지 설명할 수 있는 코드”를 만드는 것을 가장 중요하게 두었습니다.

---

## 1. 이 프로젝트를 통해 가장 먼저 공부한 것

### CRM 소프트웨어의 구조와 핵심 개념

이 프로젝트를 시작하며 가장 먼저 고민한 것은  
“CRM 소프트웨어는 왜 이렇게 설계되어야 하는가?”였습니다.

단순히 연락처·딜을 저장하는 CRUD가 아니라 아래와 같은 비즈니스 규칙이 핵심 로직이라는 점을 이해하는 데 집중했습니다.

- 리드 → 딜 전환 흐름
- 파이프라인과 스테이지 개념
- 객체(Deal / Lead / Contact / Company) 간의 관계
- 삭제 불가 조건, 상태 전이 같은 비즈니스 제약

이를 위해 실제 세일즈맵 서비스를 직접 사용하고
공식 문서를 참고하며 기능의 의도를 먼저 파악한 뒤 개발을 진행했습니다.

---

## 2. 구조 설계를 통해 공부한 것

### 도메인 단위 설계의 중요성

과제 평가 기준을 만족하기 위해 가장 중요하게 생각한 것은  
폴더 구조와 코드 구조가 도메인 단위로 명확히 드러나는가였습니다.

- API는 단순 요청 처리자가 아니라 도메인 규칙을 강제하는 계층
- 정책 판단 로직은 `src/lib`로 분리
- UI는 상태 변경의 결과를 표현하는 역할에 집중

이를 통해 다음을 학습했습니다.

- “기능 기준 분리”보다 “책임 기준 분리”가 유지보수에 유리하다는 점
- 구조가 명확하면 설명하지 않아도 코드가 의도를 드러낸다는 점

---

## 3. Next.js(App Router)를 사용하며 공부한 것

### 하나의 프로젝트에서 페이지·API·레이아웃을 관리하는 방식

Next.js App Router를 사용하며 다음을 중점적으로 학습했습니다.

- `app/` 구조에서의 라우팅 책임
- API Route와 UI가 같은 도메인 구조를 공유하는 설계
- Server Actions를 활용한 서버 중심 로직 처리

이를 통해  
페이지, API, 레이아웃을 한 프로젝트에서 일관되게 관리할 수 있었고  
설계 의도와 책임 분리를 코드 구조 자체로 드러낼 수 있었습니다.

---

## 4. Prisma + MySQL을 사용하며 공부한 것

### ORM은 편의 도구가 아니라 “설계 도구”라는 점

Prisma를 사용하면서 단순 CRUD를 넘어서 다음을 학습했습니다.

- 스키마 설계가 곧 서비스 구조라는 점
- 관계 정의(1:N, N:M)가 로직 복잡도에 직접적인 영향을 준다는 점
- 마이그레이션 히스토리가 설계 변경의 기록이라는 점

특히 아래 내용을 구현하며 DB 설계가 로직 안정성의 핵심임을 체감했습니다.

- Deal ↔ Company / Contact / Lead 관계 확장
- 리드 → 딜 전환 시 데이터 무결성 유지
- 삭제 불가 조건과 같은 비즈니스 제약 반영

---

## 5. dnd-kit을 사용하며 공부한 것

### 드래그 앤 드롭은 UI가 아니라 “상태 전이 문제”

칸반 보드 구현을 위해 dnd-kit을 사용하며 다음을 고민했습니다.

- 단순 이동이 아니라 이동 가능한 조건을 어떻게 통제할 것인가
- 순서 변경이 DB와 어떻게 동기화되어야 하는가
- 불필요한 기능이 로직 복잡도를 키우지는 않는가

dnd-kit은 필요한 동작만 조합할 수 있어  
칸반 이동 규칙을 명확하게 제어하면서도 구조를 단순하게 유지할 수 있었습니다.

---

## 6. 인증/보안을 구현하며 공부한 것

### JWT + HttpOnly 쿠키 구조

보안 구현에서 목표는 “최대한 단순하지만 안전한 구조”였습니다.

- 클라이언트에 토큰을 노출하지 않기
- 서버가 인증 책임을 명확히 가지기
- 복잡한 인증 플로우를 만들지 않기

이를 통해  
JWT는 인증 수단이지 보안의 전부가 아니라는 점과  
보안은 구조로 해결해야 한다는 점을 다시 정리할 수 있었습니다.

---

## 7. AI 도구를 활용하며 느낀 점

이 프로젝트에서는 AI 도구의 도움을 적극적으로 활용했습니다.

다만
- AI가 작성한 코드를 그대로 사용하는 것이 아니라
- 내가 이해하지 못한 부분은 반드시 재구성하거나 직접 구현했습니다.

AI를 사용해도  
개발 흐름과 로직을 놓치는 순간 더 이상 개발이 진행되지 않는다는 점을 느꼈고  
결국 “이해하지 못한 코드는 내 코드가 아니다”라는 기준을 세우게 되었습니다.

---

## 8. 아쉬운 점과 앞으로 더 공부하고 싶은 부분

- CRM 소프트웨어의 더 복잡한 실무 시나리오 이해
- 대규모 데이터 기준의 성능 최적화
- SaaS 구조에서의 멀티 워크스페이스 설계
- 정책 로직과 도메인 로직을 더 명확히 분리하는 방식

이번 과제는 축소판 CRM이지만  
전체 구조를 이해하기 위한 출발점으로 의미 있는 경험이었다고 생각합니다.
